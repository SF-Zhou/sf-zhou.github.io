# lockmap-lru：一种支持细粒度 Per-Key 锁的并发 LRU Cache

### 1. 背景

在很多业务场景中都需要 LRU Cache：热点数据缓存、连接池管理、文件句柄复用等。单线程 LRU 实现比较直观——一个 HashMap 加一条双向链表就够了。但在高并发场景下，传统的做法要么全局加锁，要么分桶加锁但粒度仍然偏粗。前者在高 QPS 时会成为瓶颈，后者虽然好一些，但同一个桶内的操作仍然需要排队。

[lockmap](https://github.com/SF-Zhou/lockmap) 是笔者写的一个 Rust 库，其中 `LruLockMap` 实现了一种 **Per-Key 级别加锁** 的并发 LRU Cache。核心思路是：不同 key 的操作完全并行，只有相同 key 的操作才需要互斥；同时每个 shard 内部维护一条侵入式双向链表来追踪 LRU 顺序。本文重点阐述其中的设计与实现。

### 2. 整体架构

`LruLockMap` 的内部结构分为三层：

```
LruLockMap
├── hasher: RandomState           // 全局唯一的哈希器
└── shards: Vec<LruShardMap>      // 多个分片
     └── inner: Mutex<LruShardInner>
          ├── table: HashTable<AliasableBox<State>>  // hashbrown 哈希表
          ├── head: *mut State    // LRU 链表头（最近访问）
          ├── tail: *mut State    // LRU 链表尾（最久未访问）
          └── max_size: usize     // 分片容量上限
```

分片数量默认为 `available_parallelism * 4` 并向上取整到 2 的幂次。每个 key 只需要做一次哈希运算，高 32 位用于路由到 shard，完整的 hash 值存储在 entry 内部、复用于 HashTable 的定位。

每个 shard 都有一把 `std::sync::Mutex`，但这把锁只用于保护 **哈希表结构和链表指针** 的修改，持有时间极短（不涉及用户数据的读写）。真正的数据读写由每个 entry 内部的 `parking_lot::RawMutex` 保护，这就是 Per-Key 锁。

### 3. Per-Key State

每个 key 在哈希表中对应一个 `State` 结构：

```rust
struct State<K, V> {
    key: K,
    hash: u64,
    flags: AtomicU32,           // 引用计数 + has_value 标记
    mutex: RawMutex,            // per-key 互斥锁
    value: UnsafeCell<Option<V>>,
    prev: UnsafeCell<*mut Self>, // 侵入式链表前驱
    next: UnsafeCell<*mut Self>, // 侵入式链表后继
}
```

几个关键设计：

**（1）`AliasableBox` 保证地址稳定。** `State` 被装在 `AliasableBox`（本质是一个不可移动的堆分配 Box）中存入 `HashTable`。这样即使 `HashTable` 扩容 rehash，`State` 的堆地址也不会变，侵入式链表的裸指针始终有效，持有 per-key 锁的线程也不会因为扩容而访问到非法地址。

**（2）`StateFlags` 将引用计数和状态位压缩在一个 `AtomicU32` 中。** 高 1 位表示当前 entry 是否有值（`has_value`），低 31 位存储引用计数（`refcnt`）。引用计数记录的是当前有多少个 `LruEntry` guard 指向该 entry。当 `refcnt == 0 && !has_value` 时，entry 就处于 `pending_cleanup` 状态，可以被安全地从哈希表中移除。

**（3）两级锁分离。** 对某个 key 进行操作时，先短暂持有 shard mutex，在哈希表中找到或插入 `State`，增加引用计数、更新链表位置，然后立即释放 shard mutex；接着锁住 `State` 内部的 per-key mutex，进入 `LruEntry` guard。此时其他 key 即使在同一个 shard 内，也可以自由地进行同样的操作。用户在 `LruEntry` guard 内对 value 的读写不会阻塞同 shard 的其他 key。

### 4. 侵入式 LRU 双向链表

每个 shard 内部维护一条双向链表，`head` 指向最近访问的 entry，`tail` 指向最久未访问的。链表的 `prev` / `next` 指针直接嵌入在 `State` 结构体中，不需要额外的节点分配。

所有链表操作（`detach`、`push_front`、`move_to_front`）都在 shard mutex 保护下完成。访问 entry 时调用 `move_to_front` 将其提升到链表头部；新插入的 entry 直接 `push_front`。因为 shard mutex 持有时间很短（只修改几个指针），对整体并发性能的影响非常小。

### 5. 非阻塞淘汰

当 shard 内的 entry 数量超过 `max_size` 时，`try_evict` 从链表尾部开始遍历，尝试淘汰最久未访问的 entry。核心逻辑如下：

```rust
fn try_evict(&mut self, current: *mut State<K, V>) {
    let mut cursor = self.tail;
    while self.table.len() > self.max_size && !cursor.is_null() && cursor != current {
        let prev = unsafe { *(*cursor).prev.get() };
        let state = unsafe { &*cursor };

        if state.flags().refcnt() > 0 {
            cursor = prev;  // 跳过正在使用的 entry
            continue;
        }

        unsafe { self.detach(cursor) };
        // 从 HashTable 中移除
        if let Ok(entry) = self.table.find_entry(state.hash, |s| std::ptr::eq(&**s, cursor)) {
            let _ = entry.remove();
        }
        cursor = prev;
    }
}
```

关键点在于：**正在被持有的 entry（`refcnt > 0`）不会被淘汰**，遍历会跳过它们继续寻找下一个候选者。这保证了在高并发场景下，即使尾部的 entry 正在被其他线程使用，淘汰过程也不会阻塞，也不会产生 use-after-free。整个淘汰过程都在 shard mutex 保护下完成。

### 6. LruEntry Guard 的生命周期

`LruEntry` 是 RAII guard，持有者对 key-value 有排他访问权。创建时的流程：

1. 哈希 key，路由到 shard；
2. 持有 shard mutex，在 `HashTable` 中查找或插入 `State`，`refcnt += 1`，`move_to_front`，调用 `try_evict`，释放 shard mutex；
3. 锁住 `State` 内部的 per-key mutex，返回 `LruEntry`。

Drop 时的流程稍微复杂一些：

1. 更新 `has_value` 标记；
2. 释放 per-key mutex；
3. CAS 循环减少引用计数——如果不是最后一个 guard、或者 entry 仍有 value，直接返回；
4. 如果是最后一个 guard 且 value 为空（`pending_cleanup`），则获取 shard mutex，再次 `dec_ref` 检查，从链表和哈希表中移除 entry。

之所以需要在 drop 时再次获取 shard mutex 做最终检查，是因为在释放 per-key mutex 和减少引用计数之间，可能有其他线程已经通过 `entry()` 或 `entry_by_ref()` 增加了引用计数。CAS + 二次检查保证了不会误删正在被其他线程使用的 entry。

### 7. 与常见实现的对比

传统并发 LRU Cache 一般有两种思路：

1. **全局锁**：简单直接，但高并发时锁争用严重。
2. **分片锁**：每个分片一把锁保护所有操作（包括数据读写），粒度取决于分片数。

`LruLockMap` 的做法介于两者之间但更加精细：shard 锁的临界区只覆盖哈希表和链表的结构修改，数据读写由 per-key 锁保护。这意味着即使两个 key 被路由到同一个 shard，它们的数据读写也可以并行。对于 value 类型较大（例如缓存一个完整的数据块）或访问时间较长（例如需要做计算或 IO）的场景，这种设计的优势更加明显。

另一个值得注意的细节是，`get` 操作在 shard mutex 内部如果发现 `refcnt == 0`，会直接 clone value 并返回，不需要获取 per-key 锁。这是一个快速路径优化：无竞争时 `get` 只需要一次短暂的 shard mutex 加锁。

### 8. 小结

[lockmap](https://github.com/SF-Zhou/lockmap) 中 `LruLockMap` 的核心设计可以归纳为：

1. **分片 + Per-Key 两级锁**，shard 锁保护结构、per-key 锁保护数据，最大化并发度；
2. **侵入式双向链表**维护 LRU 顺序，零额外分配；
3. **`AliasableBox`** 保证 `State` 地址稳定，链表裸指针和 per-key 锁始终有效；
4. **非阻塞淘汰**，跳过正在使用的 entry；
5. **`StateFlags` 原子状态压缩**，引用计数和 value 状态共享一个 `AtomicU32`，通过 CAS 实现无锁的状态转换。
